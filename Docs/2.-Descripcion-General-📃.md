## **2.1	Perspectiva del Producto**

**Notiz (Java Version)** es un sistema empresarial autónomo y modular diseñado para digitalizar, estructurar y optimizar los procesos de gestión académica en instituciones educativas. A diferencia de las metodologías tradicionales basadas en hojas de cálculo propensas a errores o sistemas heredados monolíticos, esta versión de **Notiz** implementa una arquitectura desacoplada y orientada a servicios (SOA) impulsada por **Java 21 LTS** y **Spring Boot 3.x**.

El núcleo del sistema interactúa directamente con una base de datos documental **MongoDB**, la cual proporciona un esquema dinámico y flexible sumamente adecuado para el manejo de calificaciones complejas y criterios dinámicos. Las interfaces gráficas cliente interactúan mediante una **API RESTful** segura y un canal bidireccional asíncrono soportado por **Spring WebSockets**.

## **2.2	Funciones del Producto**

El backend de **Notiz** centraliza el procesamiento de lógica de negocio organizándose en los siguientes módulos de servicio y controladores Spring:

![Image](https://github.com/user-attachments/assets/08938c35-12ab-47f8-82bb-2a7801c606ee)

**Gestión de Usuarios y Seguridad:**
* Registro e inicio de sesión local y federado utilizando tokens **JWT stateless** administrados por `Spring Security`.
* Control de acceso basado en roles (RBAC) mediante anotaciones `@PreAuthorize` para limitar accesos.
* Actualización de datos de perfil, restablecimiento seguro de contraseñas y desactivación temporal de cuentas.

**Gestión Académica:**
* Creación y mantenimiento de registros de asignaturas, periodos académicos y cursos.
* Asignación relacional lógica de profesores a asignaturas y matriculación de estudiantes a cursos específicos.
* Configuración dinámica de criterios de evaluación con pesos ponderados almacenados de forma embebida.

**Registro y Gestión de Calificaciones:**
* Carga individual de notas y carga masiva mediante procesamiento asíncrono de archivos **Excel (XLSX) y CSV** con validación sintáctica por medio de la biblioteca **Apache POI**.
* Almacenamiento flexible de observaciones de notas, retroalimentaciones y notas de recuperación.
* Gestión transaccional de peticiones de modificación de calificaciones iniciadas por los docentes.

**Consulta y Seguimiento:**
* Visualización en tiempo real de calificaciones y cálculo automatizado del promedio ponderado en la capa de servicios.
* Historial académico consolidado del estudiante a través del ciclo escolar.

**Reportes y Análisis:**
* Generación asíncrona de reportes y exportación estricta de documentos en formato **PDF/A-1a** (utilizando la biblioteca OpenPDF o similar) y hojas de cálculo.
* Visualización agregada de métricas y rendimiento del grupo académico.

**Comunicación Interactiva:**
* Mensajería asíncrona y foros grupales gestionados mediante un broker de **WebSockets con protocolo STOMP** en tiempo real.

## **2.4	Entorno Operativo**

El sistema se diseñará bajo un modelo cliente-servidor robusto que garantiza escalabilidad y compatibilidad en múltiples infraestructuras:

**En el Lado del Cliente (Frontend):**
* **Dispositivos:** Computadoras personales (Windows, macOS, Linux), tablets y smartphones.
* **Navegadores Soportados:** Versiones modernas que cumplan los estándares HTML5, CSS3 y motor JavaScript ES6+.
  - Google Chrome: versión 90 o superior.
  - Mozilla Firefox: versión 88 o superior.
  - Microsoft Edge: versión 90 o superior.
* **Conexión:** Red con un ancho de banda mínimo recomendado de 2 Mbps para cargas asíncronas de datos.

**En el Lado del Servidor (Backend & Infraestructura):**
* **Entorno de Ejecución:** Java Runtime Environment (JRE) 21 LTS.
* **Framework Backend:** Spring Boot 3.3.x (Spring Web, Spring Security, Spring Data MongoDB, Spring Cache).
* **Motor de Base de Datos Principal:** MongoDB Atlas (instancia administrada en la nube con escalamiento horizontal y réplicas de datos).
* **Motor de Base de Datos en Memoria:** Redis (para caché de metadatos, almacenamiento de tokens invalidados y soporte del broker STOMP).
* **Entorno de Despliegue:** Render.com (alojamiento en la nube mediante contenedores basados en Docker con una imagen base de Alpine Linux y JRE 21).
* **Sistema Operativo del Contenedor:** Linux Ubuntu 22.04 LTS o Alpine Linux de bajo peso.


## **2.5	Restricciones de Diseño e Implementación**

La construcción de **Notiz** bajo el stack empresarial de Java estará sujeta a las siguientes restricciones técnicas e institucionales:

**Cumplimiento Regulatorio y Privacidad:**
* **Leyes de Protección:** Cumplimiento de la **Ley 1581 de 2012 (Colombia)** y el **GDPR** europeo. Los datos sensibles (identificaciones, correos y registros académicos) deben almacenarse cifrados en reposo en MongoDB.
* **Derecho al Olvido:** Se deben proveer servicios seguros de borrado lógico y borrado físico de información personal a petición del usuario, garantizando la eliminación en cascada en las colecciones.

**Restricciones Técnicas de Desarrollo:**
* **Estructura Responsive:** La interfaz web debe diseñarse utilizando HTML5 semántico y CSS3 puro de alta fidelidad, con adaptabilidad completa y soporte para modo oscuro según las directrices WCAG 2.1 nivel AA.
* **Integración con Excel/CSV:** El parsing y generación de archivos Excel debe realizarse en el backend mediante la biblioteca **Apache POI**, y para archivos planos CSV mediante **OpenCSV**, garantizando un consumo controlado de memoria heap en el contenedor Java.

**Restricciones de Integración y Autenticación:**
* **Federación Externa:** El sistema debe integrarse exclusivamente con **Google Identity Services (Google OAuth2)** mediante el cliente de seguridad `spring-boot-starter-oauth2-client` para inicios de sesión institucionales autorizados.
* **Servicios RESTful:** Toda comunicación interactiva de datos estructurados entre el backend de Spring Boot y el cliente se realizará mediante endpoints **REST API** con formato de intercambio **JSON**.

**Restricciones de Seguridad:**
* **Protección contra Inyecciones:** Se prohíbe el uso de consultas nativas concatenadas a mano. Las operaciones con base de datos NoSQL deben gestionarse a través de interfaces parametrizadas de `MongoRepository` o mediante el uso controlado de `MongoTemplate` con enlace de variables seguro para evitar ataques de inyección de código NoSQL.
* **Seguridad de Sesiones:** Las sesiones deben ser totalmente stateless en el servidor, utilizando tokens **JWT firmados criptográficamente** (mediante algoritmos HMAC-SHA256 o superior). El cierre de sesión global de todos los dispositivos (`NZ-007`) se gestionará invalidando los tokens activos a través de una lista negra (blacklist) temporal alojada en **Redis**.
* **Auditoría Interna:** Cada acción de escritura, modificación o eliminación crítica debe disparar un registro de auditoría (`NZ-046`) de manera inmutable y asíncrona mediante listeners del ciclo de vida de Spring Data (`AfterSaveCallback`).

**Mantenimiento y Arquitectura Multitenant:**
* **Operación Monopersonal:** Al estar mantenido por un único ingeniero, la arquitectura del código Spring Boot debe organizarse rígidamente bajo la arquitectura limpia (Controladores, Servicios, Documentos, Repositorios) y usar pruebas unitarias con JUnit 5 para evitar regresiones.
* **Aislamiento Multitenant:** El sistema debe aislar lógicamente los datos de cada institución educativa mediante un identificador único de tenant (`tenantId`) incorporado en todos los documentos de MongoDB, aplicando filtros automáticos de consulta en la capa de persistencia de Spring Data.

## **2.7	Supuestos y Dependencias**

**Supuestos Organizacionales e Infraestructura:**
* **Conectividad:** Se asume que las instituciones educativas contratantes dispondrán de una conexión a internet estable a nivel institucional para el personal docente y administrativo.
* **Capacidad del Usuario:** Se presupone que el profesorado cuenta con un dominio básico en el manejo de archivos de hojas de cálculo (.xlsx, .csv) y de navegadores web modernos.
* **Volumen Académico:** El diseño arquitectónico asume una carga por tenant de entre 500 y 1000 alumnos concurrentes en periodos de alto tráfico escolar.

**Dependencias Técnicas Externas:**
* **Google Identity Provider:** Dependencia de la disponibilidad del servidor de autorización OAuth2 de Google para el proceso federado de logeo de usuarios.
* **SendGrid API:** Dependencia en el proveedor de correos para el envío asíncrono de alertas de notas y tokens de doble factor por correo.
* **Entorno de Contenedores:** Dependencia del correcto funcionamiento de la plataforma de contenedores Docker de Render.com y de la base de datos documental administrada de MongoDB Atlas.
