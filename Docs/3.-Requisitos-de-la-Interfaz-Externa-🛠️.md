## **3.1	Interfaces de Usuario**

El diseño y representación visual de la interfaz gráfica de **Notiz (Java Version)** se basa en los style tiles creados en Canva, los cuales establecen un sistema de diseño consistente, moderno y responsive. La paleta de colores está diferenciada estratégicamente por roles de usuario para mejorar la retroalimentación visual de contexto. Se adopta la tipografía **Roboto** como fuente tipográfica principal del sistema a través de Google Fonts, garantizando legibilidad óptima y accesibilidad en múltiples resoluciones. 

La interfaz prioriza la usabilidad y los criterios del atributo de **aprendizabilidad** (`NZ-051`), proporcionando una jerarquía visual clara, contrastes legibles que cumplen con el contraste mínimo de **4.5:1 (WCAG 2.1 AA)** en su modo claro y oscuro, y micro-animaciones en CSS3 para suavizar las transiciones de carga asíncrona de notas.

## **3.2	Hardware Interfaces**

El sistema Notiz es una aplicación web multiplataforma accesible mediante navegadores estándar, por lo cual no requiere de interfaces o controladores directos de hardware específico en el lado del cliente. Sin embargo, se especifican las siguientes directrices y compatibilidades de hardware:

**Dispositivos de Entrada/Salida Soportados:**
* **Computadores Personales y Portátiles:** Dispositivos basados en arquitecturas x86/ARM ejecutando sistemas operativos modernos (Windows, macOS, Linux).
* **Dispositivos Móviles (Tablets y Smartphones):** Pantallas con resoluciones de pantalla variables (desde 320px de ancho), equipadas con entrada táctil (pantallas multitouch).
* **Impresoras y Generación:** Dispositivos con capacidades estándar de visualización de gráficos y rendering para la visualización correcta de archivos vectoriales y la descarga física de documentos en formato de conservación a largo plazo **PDF/A-1a**.

La arquitectura distribuida en la nube se ejecuta en contenedores Linux gestionados por Render.com, aislando por completo al usuario de complejidades de hardware de servidor.


## **3.3	Software Interfaces**

El backend de **Notiz** (desarrollado en Spring Boot 3.3.x) interactúa con componentes, bibliotecas de software y proveedores SaaS de forma segura a través de interfaces bien definidas:

**1. Proveedor de Autenticación Federada (Google Identity Services):**
* **Descripción:** Permite el inicio de sesión federado único (SSO) de la comunidad educativa usando sus cuentas institucionales de Google.
* **Componente:** Integrado en la cadena de filtros de `Spring Security` a través del cliente `spring-boot-starter-oauth2-client`.
* **Protocolo:** OpenID Connect (OIDC) / OAuth 2.0. La aplicación Java intercambia de forma segura los códigos de autorización por tokens de identidad (ID Tokens) firmados por los servidores de Google para autenticar la identidad del usuario y mapear sus roles institucionales.

**2. Base de Datos Documental (MongoDB Driver):**
* **Descripción:** Capa de comunicación directa para operaciones NoSQL de lectura, escritura y agregación.
* **Componente:** `mongodb-driver-sync` e interfaces dinámicas de `Spring Data MongoDB`.
* **Mapeo:** Uso de anotaciones de Spring Data (como `@Document`, `@Id`, `@Field`, `@Indexed`) para mapear de manera fluida clases Java (POJOs) a documentos estructurados BSON dentro de colecciones de MongoDB Atlas.

**3. API de Correo Transaccional (SendGrid SDK):**
* **Descripción:** Envío asíncrono y seguro de notificaciones automáticas de notas, alertas de seguridad, registros de usuarios y códigos de verificación en dos factores (2FA).
* **Componente:** `sendgrid-java` SDK o `spring-boot-starter-mail` configurado mediante transporte SMTP/S.
* **Seguridad:** Autenticación por HTTPS mediante llave de API (*API Key*) y cifrado TLS 1.3 de extremo a extremo, resguardando las llaves en variables de entorno inyectadas en Render.com.

**4. Capa de Gestión de Caching y Sesiones (Spring Data Redis):**
* **Descripción:** Conexión con un clúster Redis para caché de consultas académicas costosas y validación de tokens JWT revocados.
* **Componente:** Biblioteca cliente `Lettuce` integrada con `Spring Cache` para optimizar los accesos concurrentes de bases de datos.


## **3.4	Interfaces de Comunicación**

La interoperabilidad y el flujo bidireccional seguro de datos entre las interfaces cliente y el backend de Spring Boot se rige bajo los siguientes estándares de comunicación:

**1. Protocolo de Red de Extremo a Extremo:**
* **Estándar:** **HTTPS (Hypertext Transfer Protocol Secure)**.
* **Cifrado:** Negociación obligatoria de **TLS 1.2 o TLS 1.3**. Quedan prohibidas todas las comunicaciones no seguras por puerto HTTP plano (redireccionamiento automático 301 forzado en el balanceador de Render.com).

**2. Formato de Datos de la API REST:**
* **Estándar:** **JSON (JavaScript Object Notation)**.
* **Estructura:** Datos estructurados serializados y deserializados en el backend de Spring Boot mediante la biblioteca **Jackson** integrada en Spring MVC. Las llamadas asíncronas de la interfaz cliente (Fetch API / Axios) envían y reciben recursos bajo este estándar.

**3. Formatos de Salida de Archivos y Reportes:**
* **Formatos Soportados:** Hojas de cálculo Excel (**XLSX**) y documentos planos de texto (**CSV**) generados dinámicamente mediante el procesamiento de flujos de bytes.
* **Generación de Reportes:** Documentos **PDF** con cumplimiento de la norma internacional **PDF/A-1a** (ISO 19005-1) utilizando la biblioteca **OpenPDF**, garantizando la legibilidad a largo plazo e inclusión de metadatos de accesibilidad.

**4. Canal de Comunicación Asíncrona en Tiempo Real:**
* **Tecnología:** **WebSockets**.
* **Protocolo de Subcapa:** **STOMP (Simple Text Oriented Messaging Protocol)** con soporte de contingencia mediante **SockJS** para navegadores antiguos.
* **Ruta del Endpoint:** `/ws-notiz`.
* **Canales de Suscripción:** Tópicos dedicados `/topic/notifications/*` para alertas instantáneas de calificaciones registradas y canales `/queue/messages/*` para la mensajería instantánea alumno-profesor.
